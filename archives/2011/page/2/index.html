几个编码习惯的改变

JS的编码习惯：

1. 使用单引号来写字符串

原来一直是遗传了java中的写法用双引号来写字符串，但是在Javascript中使用双引号有个缺点。

[js]

var html = "<div class=\"block\"></div>";

var html = "<div class=’block’></div>";

[/js]

如果用string来表示html，那么需要在html里面写单引号。但是在一般的html写法中习惯使用了双引号。必须要用这两种方法中的一种，但是第一种会多出多余的字符，第二种方法会导致js中的html和html文件中的写法不一样。所以最近改变了习惯，用单引号写字符串。

CSS的编码习惯

1. CSS的选择符使用”-“来分割

之前的文章“更好的组织CSS”提到过用模块化的CSS选择符。

[css]

.side_login{ width:260px;overflow:hidden;}

[/css]

之前有段时间一直使用了下划线’_’来分割不同的“域”，但是实际上下划线并不是用来分割单词的，如果一个单词内部还有划分就找不到合适的符号。所以我使用了中线来代替，下划线用来分割一个单词内部的两个不同语义。例如：

[css]

.side-login_box{ width:260px;overflow:hidden;}

[/css]

另外一个好处在于编辑器方面：如果使用中线来分割单词，当你双击单词时，自动选中的是其中的一部发。但是如果是下划线会自动选中全部。在搜索时，往往是要搜一个“域”（即单词中的一部分，因为那部分是最容易重复的）而不是全部。

2. 使用class做选择符而不是html tag

先看如下例子：

[html]
<div class="control">
<span>test1</span>
<a href="#">a1</a>
<span>test2</span>
<span>test3</span>
</div>
[/html]

这样的html要写CSS的话，得到如下结果：

[css]
.control span{color:red;font-size:12px;}
.control a{color:blue;font-size:12px;}
[/css]

对于这样很容易修改的网页部分需求很容易改变。如果要添加了一个新的部分或者是只修改最后一个span，那么需要修改CSS：

[css]
.control span{color:red;font-size:12px;}
.control a{color:blue;font-size:12px;}
.control-last{color:yellow;font-size:14px;}
[/css]

单单只有这样一个修改很容易，但是如果span有很多，修改的部分也很多，那么会导致很多的冗余，例如：

[css]
.control span{color:red;font-size:12px;border:1px solid #000;background:url(imgs/add.png);}
.control a{color:blue;font-size:12px;}
.control-last{color:yellow;font-size:14px;border:none;background:none;}
.control-first{color:yellow;font-size:13px;border:none;background:none;}
.control-second{color:yellow;font-size:15px;border:none;background:none;}
[/css]

可以发现每个特殊样式会有很多的冗余：border和background。并且这样也会很容易在修改添加的时候会出现一项不到的bug。
所以说，不要给html tag的选择符添加无法继承的样式，类似与border,background这样的样式需要设置重置。如果是可以继承的样式那么倒是没什么关系。

3. 使用tab来表示CSS中的作用域

在CSS中没有原生的作用域支持，并且更重要的是编辑器不会提供代码折叠。这在修改重构CSS的样式会很难找到对应的选择符。
所以不如自己用tab来表示作用域，例子如下：
[css]
.control{width:260px;overflow:hidden;}
    .control span{color:red;font-size:12px;border:1px solid #000;background:url(imgs/add.png);}
    .control a{color:blue;font-size:12px;}
    .control-last{color:yellow;font-size:14px;border:none;background:none;}
    .control-first{color:yellow;font-size:13px;border:none;background:none;}
    .control-second{color:yellow;font-size:15px;border:none;background:none;}
[/css]

 这样虽然没有折叠，但是也可以达到一定程度的达到帮助浏览代码的目的。

4. 以内容本质来起class名字，并使用组合方式表示相同样式，而不是给它们一个相同的class

当多个不同的元素拥有多个相同的样式的时候，以前我的做法是给一个相同的class(以样式来命名)，然后统一设置。后来发现这样的可维护性很差，因为虽然暂时这多个元素具有相同的样式，但是实际上这几个元素的内容是不同！以后在进一步的改版过程中，因为内容不同所以本质就不同，导致了这几个样式可能会有不同的变化，此时要么用新的class替换旧的，要么就添加一个新的class。两者的维护性都不好。所以不如一开始就以内容本质来命名class，然后用组合的方式写CSS。如下： [css]
.side-article,
.side-ad,
.side-name,
.side-ask{
margin-bottom:10px;
width:260px;
overflow:hidden;
}
[/css]

前端组件的构建

大多数前端新人（至少不久之前我也是这么想的）对于前端组件的定义都认为是JavaScript组件，CSS只是一个皮肤，而HTML由JS动态生成。个人认为这个构建思路有点过时。

前端是由多中语言组成的，不仅仅是JS。从HMTL5的发展来看，CSS添加了动画等功能，HTML的标签更语义化。套用标准的MVC结构：CSS是View，html是model，JS则承担了一个逻辑控制的作用：control。所以说一个前端组件大概构成如下：

  1. CSS样式文件
  2. html文件
  3. js文件
    貌似这和现有的那些组件差不多，其实是一样的。看你是如何对待了。你也可以把html与CSS都看成是这个JS库的demo，这就是之前所说的：前端组件都是JS组件的情况了… 虽然实际使用这个组件的时候你很可能把demo中的html和CSS都拷贝过来！

闲话少说如何利用之前的想法构建一个“前端组件”呢？

举一个链接hover上去滑动的例子：nudging组件。

CSS3的transition样式可以提供过度效果。

同时低级浏览器不支持这个样式，那么需要用JS去实现。

这是我们就得到如下的代码：

JS：

[js]
G.def( ‘nudging’, function() {
var style = document.createElement(‘a’).style,
supportAnimate = false,
prefix = [‘-o-‘,’-webkit-‘,’-moz-‘,’-ms-‘];
for ( var i=0; i < prefix.length; i++ ) {
if ( style[ prefix[i] + ‘transition’ ] !== undefined ) {
supportAnimate = true;
}
}
/*

 * 链接放上后滑动
 */
var nudging = $.fn.nudging = function( /*@dom/selector/jquery obj*/dom ) {
    var $this = dom ? $(dom) : this;
    $this.hover(function() { //mouse in
        $(this).animate({ paddingLeft: '20px' }, 400);
    }, function() { //mouse out
        $(this).animate({ paddingLeft: 0 }, 400);
    });        
}

return {
    nudging:nudging,
    cssSupport:supportAnimate
};

});
[/js]

html:
[html]
<ul>
<li><a href="#" class=’nudging’>Link 1</a></li>
<li><a href="#" class=’nudging’>Link 2</a></li>
<li><a href="#" class=’nudging’>Link 3</a></li>
<li><a href="#" class=’nudging’>Link 4</a></li>
<li><a href="#" class=’nudging’>Link 5</a></li>
</ul>
[/html]

css:
[css]
.nudging {
cursor: pointer;
-webkit-transition: padding-left 250ms ease-out;
-moz-transition: padding-left 250ms ease-out;
transition: padding-left 250ms ease-out;
}
.nudging:hover {
padding-left: 10px;
}
[/css]

demo

主要的思路是：如果CSS transition不支持则使用js实现。

这个组件提供了一个API：nudging。这里的API不仅仅是JS的nudging方法，还有CSS class：nudging！这个组件使用的module没有限制。那么依据这些，我们可以写出这个组件API文档：

API:

  1. function nudging：使用JS实现链接的nudging
  2. var cssSupport：判断浏览器是否支持CSS transition
  3. CSS class nudging：使支持CSS transition的浏览器的链接有budging效果
    HTML Module:无限制

同样的方法可能实现很多这种组合性质的“前端组件”。

正确使用CSS API

或许有人会问，为什么不直接使用JS来修改style，达到动画效果呢？这样代码全都在js中，方便拷贝！

首先这个效果很简单，但是如果是多个元素构成的复杂效果，我们需要使用JS多次操作dom。有点前端开发经验的都知道dom操作就是前端开发的瓶颈所在！

其次不利于统一的管理，当使用JS来操作UI时。我们就需要将操作UI的代码单独独立成一个函数，函数可能被复用。如果你要新建出一种效果，这要新建一个函数。久而久之，代码中充满了UI的操作函数，和其他逻辑函数混杂在一起。或许你可以起一个好听的名字xxxUI等等，但是我们可以更加优雅的用JS来调用Class API来实现：

[js] $(dom).addClass( ‘nudging’ ); [/js]

最后谈一下：何时生成html

有了HTML Module并不代表着组件中失去了“生成html”这一过程，比如：头像剪切组件。很多的UI是在选中并上传头衔之后才会出现，也就是说一开始显示并不需要，而且不一定会使用。这是将HTML临时生成最为合适！

不过这并不是让把这些html模型嵌入到JS中，模型和控制还是要分开。可以把html模型放到demo中的JS，这样就可以起到分离的作用。

总结

最重要的一点是：控制、显示、模型三者分离，抛弃后台开发程序员时候的一种语言定天下的思路：前端组件不仅仅是JS。虽然目前在显示方面，大部分的动画效果还需要JS来实现，不过随着CSS3的普及，显示将逐渐归为CSS处理！

当然这也不是没有确定，代码的分离必然造成了“拷贝”的不方便。这就需要一个好的代码管理和合并机制，这样就只需要

[css]
@import nudgin.css
@import nudgin.js
[/css]

这样简单的语法糖就可以实现目标了。

JS读取本地的input:file路径的测试

代码如下：

[html]
<!DOCTYPE HTML>
<html lang="en-US">
<head>
<meta charset="UTF-8">
<title></title>
</head>
<body>
<input type="file" name="file" id="file" />
<img src="" alt="预览" title="预览" />
<script type="text/javascript" src="jquery.js"></script>
<script type="text/javascript">
$(function() {
var path,
clip = $("img"),
FileReader = window.FileReader;
alert(!!FileReader);
$("#file").change(function() {
if (FileReader) {
var reader = new FileReader(),
file = this.files[0];
reader.onload = function(e) {
clip.attr("src", e.target.result);
};
reader.readAsDataURL(file);
}
else {
path = $(this).val();
if (/"\w\W"/.test(path)) {
path = path.slice(1,-1);
}
alert(path);
clip.attr("src",path);
alert(clip[0].src);
}
});
});
</script>
</body>
</html>
[/html]

代码主要做了读取input:file的路径并显示本地的图片。最终测试结果：

IE 6

通过获取input:file的src值可以获取路径，并设置img的src值。

结果：显示成功！

IE 7/8

获取input:file的src值是fake值（c:\fakepath\filename.jpg）。（注意线上测试得到这种结果，但是如果是本地的html则正常）

结果：显示失败！

Firefox 3.6/Chrome 8

故获取input:file的src值只有文件名！故通过FileReader API获取image的data URI，并设置img的src。（注意：chrome浏览器下html在本地则获取不到image的dataURI）

结果：显示成功！

opera 10

不支持FileReader API，故获取input:file的src值是fake值（c:\fakepath\filename.jpg）。

结果：显示失败！
