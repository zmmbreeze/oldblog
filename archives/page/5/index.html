几个编码习惯的改变

JS的编码习惯：

1. 使用单引号来写字符串

原来一直是遗传了java中的写法用双引号来写字符串，但是在Javascript中使用双引号有个缺点。

[js]

var html = "<div class=\"block\"></div>";

var html = "<div class=’block’></div>";

[/js]

如果用string来表示html，那么需要在html里面写单引号。但是在一般的html写法中习惯使用了双引号。必须要用这两种方法中的一种，但是第一种会多出多余的字符，第二种方法会导致js中的html和html文件中的写法不一样。所以最近改变了习惯，用单引号写字符串。

CSS的编码习惯

1. CSS的选择符使用”-“来分割

之前的文章“更好的组织CSS”提到过用模块化的CSS选择符。

[css]

.side_login{ width:260px;overflow:hidden;}

[/css]

之前有段时间一直使用了下划线’_’来分割不同的“域”，但是实际上下划线并不是用来分割单词的，如果一个单词内部还有划分就找不到合适的符号。所以我使用了中线来代替，下划线用来分割一个单词内部的两个不同语义。例如：

[css]

.side-login_box{ width:260px;overflow:hidden;}

[/css]

另外一个好处在于编辑器方面：如果使用中线来分割单词，当你双击单词时，自动选中的是其中的一部发。但是如果是下划线会自动选中全部。在搜索时，往往是要搜一个“域”（即单词中的一部分，因为那部分是最容易重复的）而不是全部。

2. 使用class做选择符而不是html tag

先看如下例子：

[html]
<div class="control">
<span>test1</span>
<a href="#">a1</a>
<span>test2</span>
<span>test3</span>
</div>
[/html]

这样的html要写CSS的话，得到如下结果：

[css]
.control span{color:red;font-size:12px;}
.control a{color:blue;font-size:12px;}
[/css]

对于这样很容易修改的网页部分需求很容易改变。如果要添加了一个新的部分或者是只修改最后一个span，那么需要修改CSS：

[css]
.control span{color:red;font-size:12px;}
.control a{color:blue;font-size:12px;}
.control-last{color:yellow;font-size:14px;}
[/css]

单单只有这样一个修改很容易，但是如果span有很多，修改的部分也很多，那么会导致很多的冗余，例如：

[css]
.control span{color:red;font-size:12px;border:1px solid #000;background:url(imgs/add.png);}
.control a{color:blue;font-size:12px;}
.control-last{color:yellow;font-size:14px;border:none;background:none;}
.control-first{color:yellow;font-size:13px;border:none;background:none;}
.control-second{color:yellow;font-size:15px;border:none;background:none;}
[/css]

可以发现每个特殊样式会有很多的冗余：border和background。并且这样也会很容易在修改添加的时候会出现一项不到的bug。
所以说，不要给html tag的选择符添加无法继承的样式，类似与border,background这样的样式需要设置重置。如果是可以继承的样式那么倒是没什么关系。

3. 使用tab来表示CSS中的作用域

在CSS中没有原生的作用域支持，并且更重要的是编辑器不会提供代码折叠。这在修改重构CSS的样式会很难找到对应的选择符。
所以不如自己用tab来表示作用域，例子如下：
[css]
.control{width:260px;overflow:hidden;}
    .control span{color:red;font-size:12px;border:1px solid #000;background:url(imgs/add.png);}
    .control a{color:blue;font-size:12px;}
    .control-last{color:yellow;font-size:14px;border:none;background:none;}
    .control-first{color:yellow;font-size:13px;border:none;background:none;}
    .control-second{color:yellow;font-size:15px;border:none;background:none;}
[/css]

 这样虽然没有折叠，但是也可以达到一定程度的达到帮助浏览代码的目的。

4. 以内容本质来起class名字，并使用组合方式表示相同样式，而不是给它们一个相同的class

当多个不同的元素拥有多个相同的样式的时候，以前我的做法是给一个相同的class(以样式来命名)，然后统一设置。后来发现这样的可维护性很差，因为虽然暂时这多个元素具有相同的样式，但是实际上这几个元素的内容是不同！以后在进一步的改版过程中，因为内容不同所以本质就不同，导致了这几个样式可能会有不同的变化，此时要么用新的class替换旧的，要么就添加一个新的class。两者的维护性都不好。所以不如一开始就以内容本质来命名class，然后用组合的方式写CSS。如下： [css]
.side-article,
.side-ad,
.side-name,
.side-ask{
margin-bottom:10px;
width:260px;
overflow:hidden;
}
[/css]

前端组件的构建

大多数前端新人（至少不久之前我也是这么想的）对于前端组件的定义都认为是JavaScript组件，CSS只是一个皮肤，而HTML由JS动态生成。个人认为这个构建思路有点过时。

前端是由多中语言组成的，不仅仅是JS。从HMTL5的发展来看，CSS添加了动画等功能，HTML的标签更语义化。套用标准的MVC结构：CSS是View，html是model，JS则承担了一个逻辑控制的作用：control。所以说一个前端组件大概构成如下：

  1. CSS样式文件
  2. html文件
  3. js文件
    貌似这和现有的那些组件差不多，其实是一样的。看你是如何对待了。你也可以把html与CSS都看成是这个JS库的demo，这就是之前所说的：前端组件都是JS组件的情况了… 虽然实际使用这个组件的时候你很可能把demo中的html和CSS都拷贝过来！

闲话少说如何利用之前的想法构建一个“前端组件”呢？

举一个链接hover上去滑动的例子：nudging组件。

CSS3的transition样式可以提供过度效果。

同时低级浏览器不支持这个样式，那么需要用JS去实现。

这是我们就得到如下的代码：

JS：

[js]
G.def( ‘nudging’, function() {
var style = document.createElement(‘a’).style,
supportAnimate = false,
prefix = [‘-o-‘,’-webkit-‘,’-moz-‘,’-ms-‘];
for ( var i=0; i < prefix.length; i++ ) {
if ( style[ prefix[i] + ‘transition’ ] !== undefined ) {
supportAnimate = true;
}
}
/*

 * 链接放上后滑动
 */
var nudging = $.fn.nudging = function( /*@dom/selector/jquery obj*/dom ) {
    var $this = dom ? $(dom) : this;
    $this.hover(function() { //mouse in
        $(this).animate({ paddingLeft: '20px' }, 400);
    }, function() { //mouse out
        $(this).animate({ paddingLeft: 0 }, 400);
    });        
}

return {
    nudging:nudging,
    cssSupport:supportAnimate
};

});
[/js]

html:
[html]
<ul>
<li><a href="#" class=’nudging’>Link 1</a></li>
<li><a href="#" class=’nudging’>Link 2</a></li>
<li><a href="#" class=’nudging’>Link 3</a></li>
<li><a href="#" class=’nudging’>Link 4</a></li>
<li><a href="#" class=’nudging’>Link 5</a></li>
</ul>
[/html]

css:
[css]
.nudging {
cursor: pointer;
-webkit-transition: padding-left 250ms ease-out;
-moz-transition: padding-left 250ms ease-out;
transition: padding-left 250ms ease-out;
}
.nudging:hover {
padding-left: 10px;
}
[/css]

demo

主要的思路是：如果CSS transition不支持则使用js实现。

这个组件提供了一个API：nudging。这里的API不仅仅是JS的nudging方法，还有CSS class：nudging！这个组件使用的module没有限制。那么依据这些，我们可以写出这个组件API文档：

API:

  1. function nudging：使用JS实现链接的nudging
  2. var cssSupport：判断浏览器是否支持CSS transition
  3. CSS class nudging：使支持CSS transition的浏览器的链接有budging效果
    HTML Module:无限制

同样的方法可能实现很多这种组合性质的“前端组件”。

正确使用CSS API

或许有人会问，为什么不直接使用JS来修改style，达到动画效果呢？这样代码全都在js中，方便拷贝！

首先这个效果很简单，但是如果是多个元素构成的复杂效果，我们需要使用JS多次操作dom。有点前端开发经验的都知道dom操作就是前端开发的瓶颈所在！

其次不利于统一的管理，当使用JS来操作UI时。我们就需要将操作UI的代码单独独立成一个函数，函数可能被复用。如果你要新建出一种效果，这要新建一个函数。久而久之，代码中充满了UI的操作函数，和其他逻辑函数混杂在一起。或许你可以起一个好听的名字xxxUI等等，但是我们可以更加优雅的用JS来调用Class API来实现：

[js] $(dom).addClass( ‘nudging’ ); [/js]

最后谈一下：何时生成html

有了HTML Module并不代表着组件中失去了“生成html”这一过程，比如：头像剪切组件。很多的UI是在选中并上传头衔之后才会出现，也就是说一开始显示并不需要，而且不一定会使用。这是将HTML临时生成最为合适！

不过这并不是让把这些html模型嵌入到JS中，模型和控制还是要分开。可以把html模型放到demo中的JS，这样就可以起到分离的作用。

总结

最重要的一点是：控制、显示、模型三者分离，抛弃后台开发程序员时候的一种语言定天下的思路：前端组件不仅仅是JS。虽然目前在显示方面，大部分的动画效果还需要JS来实现，不过随着CSS3的普及，显示将逐渐归为CSS处理！

当然这也不是没有确定，代码的分离必然造成了“拷贝”的不方便。这就需要一个好的代码管理和合并机制，这样就只需要

[css]
@import nudgin.css
@import nudgin.js
[/css]

这样简单的语法糖就可以实现目标了。

DSL与函数式编程

什么是DSL？

DSL即领域编程语言，它是用于解决特定领域问题的语言。与GPPL（通用目的编程语言）相比，DSL针对的目标是特定的领域。其实DSL并不是什么高深的东西，SQL、正则表达式、CSS等等都是一种DSL。对于前端工程师来说jQuery就是一种DSL。

DSL可以分为两种：内部DSL、外部DSL。外部DSL是自我包含的语言，它们有自己特定语法、解析器和词法分析器等等，它往往是一种小型的编程语言，甚至不会像GPPL那样需要源文件。与之相对的则是内部DSL。内部DSL其实更像是种别称，它代表一类特别API及使用模式[1]。jQuery就是一种内部DSL，而CSS则是一种外部DSL。

[js]$(".selector").text("display text")
.append("<div>children</div>")
.show();[/js]

使用DSL的好处就是：我们可以更关注做什么（What）而不是怎么做（How）!
jQuery是一门用于解决多种浏览器上javascript 执行兼容性的DSL，它提供了一组API用于Web前端开发的通用功能：DOM、AJAX、Event等等。众所周知，浏览器的兼容性是前端开发中最难解决的问题，而使用jQuery可以避免多数bad parts，只使用Good parts。这样我们在开发时就可以更关注做什么（What），而不用管底层的浏览器js是怎么做（how）的。

函数式编程

编程语言的进化史是这样的：

  1. 打孔机，机器语言（记得操作系统课上老师展示过…）
  2. 汇编语言：最底层的操作都通过代码来指定
  3. 面向过程语言：Picasal、C
  4. 面向对象语言：C++
  5. 虚拟机时代（垃圾收集、类型安全）：Java、C#
  6. 动态语言：Python、Ruby
    扯的有点远了，我想说的是：编程语言越来越抽象了…

在数学中函数的抽象定义是元素(输入)与元素（输出）之间的对应数学关系。例如：y=f(x)定义了x与y之间的数学关系，函数计算完成之后x的值不会改变，并返回一个输出y。

在编程语言中为了重复利用一些特定操作，于是产生了函数。函数被定义为：根据输入计算得到输出的一批程序指令。函数计算完成之后输入可能会改变，并返回一个输出或不返回。

编程语言中的函数与数学函数最大的区别在于：函数是有状态的，它可以修改外部变量或输入变量。而数学函数没有状态、没有变量。并且程序语言的函数除了计算值之外还可以做些额外的操作。这些操作可以是：打印日志、调整显示GUI等等。

函数式编程的定义是：将计算等同于数学函数的编程范式。“函数式编程”中的函数是数学函数而不是一般程序语言中的函数。

函数式编程与一般的命令式编程的区别：

[js]//命令式编程：
function Counter(_i) {
var i = _i || 0;
this.add = function(j) {
i = i+j;
}
}
//函数式编程：
function Counter(_i) {
var i = _i || 0;
this.add = function(j) {
return new Counter(i+j);
}
}[/js]

两者的区别在于函数式编程的add方法并没有修改i变量，也就是Counter没有可变的状态。这赋予了函数式编程最大的优点：线程安全。虽然前端的javascript开发并没有涉及到多线程的情况，不过随着HTML5的普及，WebWorker的出现也会使的javascript渐渐出现多线程并行问题。

当然函数式编程的优点并不只在于线程安全这一个，而且更在于它在某些情况下的结构清晰，例如：修改给所有数组中的值加1。

[js]
//命令式编程：
function add(array) {
for (var i=0,length=array.length;i&lt;length;i++)
array[i]++;
}
//函数式编程：
function add(n) { return n+1; } //不修改输入变量
_(array).map(add);
[/js]

两端段代码，相比之下后者的结构更加清晰，因为它相比命令式编程更进一步抽象了。函数式编程达到了和DSL一样的目的：我们可以更关注做什么（what）而不是怎么做（how），提供更清晰的代码结构。除了例子中map之外，还有延迟计算、Memorization等等这些都是函数式编程中的Good parts。

之前那段代码中用到了一个javascript库就是understandcore.js，这个库提供了在js中实现函数式编程的功能。当然做到完全的使用函数式编程是不可能的！

  1. 因为我们总会遇到需要在函数中做其他操作的情况，例如打印日志；
  2. 还有性能问题，第一个例子中函数式编程明显在性能上有很大劣势；
    所以“函数式编程”也需要在适当的时候使用才能发挥作用，不能滥用！

更好的组织CSS

一开是你的代码是这样的：

[html]index.html
<div>
<span class="title">标题</span><p>共<span class="counter">34</span>此评论</p>
<p>内容！</p>
</div>

article_list.html:
<ul>
<li><h2>标题</h2><p>内容简介！</p></li>
<li><h2>标题</h2><p>内容简介！</p></li>
</ul>

g.css:
p{color:#000;font-size:12px;}
h2{color:blue;font-size:24px;margin:10px 20px;}
.title{color:blue;font-size:24px;margin:10px 20px;}
.counter{color:blue;font-size:14px;padding:0 10px;}
[/html]

这样的代码即不方便维护也不方便查看。于是下定决心开始重构！你想到了代码的模块化，解决方法如下：

[html]
<div class="entry">
<span class="title">标题</span><p>共<span class="counter">34</span>此评论</p>
<p>内容！</p>
</div>

article_list.html:
<ul class="list">
<li><h2>标题</h2><p>内容简介！</p></li>
<li><h2>标题</h2><p>内容简介！</p></li>
</ul>

g.css:
.entry p,.list p{color:#000;font-size:12px;}
.list h2,.entry .title{color:blue;font-size:24px;margin:10px 20px;}
.entry .counter{color:blue;font-size:14px;padding:0 10px;}
[/html]

这样方便维护并且页便于查看，但是仔细一想。为了要在一个选择符中用两个class，完全可以合并成一个，这样效率更高：

[html]
<div class="entry">
<span class="entry_title">标题</span><p>共<span class="entry_counter">34</span>此评论</p>
<p>内容！</p>
</div>

article_list.html:
<ul class="list">
<li><h2>标题</h2><p>内容简介！</p></li>
<li><h2>标题</h2><p>内容简介！</p></li>
</ul>

g.css:
.entry p,.list p{color:#000;font-size:12px;}
.list h2,.entry_title{color:blue;font-size:24px;margin:10px 20px;}
.entry_counter{color:blue;font-size:14px;padding:0 10px;}
[/html]

如此，你可以通过模块名作为class的namespace来实现代码的易维护和易查看。这样就可以了吗？如果代码结构变得越来越复杂，如下：
[html]
<div class="entry">
<span class="entry_title">标题</span>
<p>共<span class="entry_counter">34</span>此评论&nbps;作者：<a href="#" class="entry_author">XXX</a></p>
<p>内容！</p>
</div>

article_list.html:
<h1><a href="#">XXX</a></h1>
<ul class="list">
<li><h2>标题</h2><p>内容简介！</p></li>
<li><h2>标题</h2><p>内容简介！</p></li>
</ul>

g.css:
.entry p,.list p{color:#000;font-size:12px;}
.list h2,.entry_title{color:blue;font-size:24px;margin:10px 20px;}
.entry_counter{color:blue;font-size:14px;padding:0 10px;}
.entry_author{color:blue;}
a{color:#000;}
[/html]
这样代码又变得复杂了，如果有一天类似于国殇日那样的情况发生，整个网站的蓝色文字需要变成黑色的！那么你就有的改了…
那么有什么办法让它变得更加易维护和易查看。学过设计模式的都知道：“组合优于继承”。虽然CSS没有继承，但是我们可以用组合使模块之间的关联更少。显然我们可以把颜色部分单独抽取出来，作为一个复用的类！

  1. 将重复复用的CSS提取出一个额外的class；
  2. 模块中的类名倾向于内容的含义，例如：title;
  3. 复用的类名倾向于表现的含义，例如：blue;
    这样得到了如下代码：
    [html]
    <div class="entry">
    <span class="entry_title big">标题</span>
    <p>共<span class="entry_counter middle blue">34</span>此评论&nbps;作者：<a href="#" class="blue">XXX</a></p>
    <p>内容！</p>
    </div>

article_list.html:
<h1><a href="#">XXX</a></h1>
<ul class="list">
<li><h2 class="big blue">标题</h2><p>内容简介！</p></li>
<li><h2 class="big blue">标题</h2><p>内容简介！</p></li>
</ul>

g.css:
//common
.blue{color:blue;}
.big{font-size:24px;}
.middle{font-size:14px;}
//module
.entry p,.list p{color:#000;font-size:12px;}
.list h2,.entry_title{margin:10px 20px;}
.entry_counter{padding:0 10px;}
a{color:#000;}
[/html]
这样为止，看起来还不错！易维护和可查看性都大大提高了。唯一的缺点是class增加了….不过这些损失换来的好处是值得的。
如今流行的CSS框架也是走样做的，例如：blueprint。
如果还想进一步组织，那么你就可以使用grid系统，例如960 Grid System。不过这需要与设计配合，所以不是由你决定的。

但是有一个缺点让我始终存在心病：“class太多了”。那到底有什么好的办法呢？本人刚从第五届D2归来，这次D2上出现了N多的Javascript编译工具，可以让你用更好的代码风格来写Javascript。同样这也适用于CSS，看这个LESS CSS。之前的问题被迅速解决了，而且没有增加class：
[html]
<div class="entry">
<span class="entry_title">标题</span>
<p>共<span class="entry_counter">34</span>此评论&nbps;作者：<a href="#" class="blue">XXX</a></p>
<p>内容！</p>
</div>

article_list.html:
<h1><a href="#">XXX</a></h1>
<ul class="list">
<li><h2>标题</h2><p>内容简介！</p></li>
<li><h2>标题</h2><p>内容简介！</p></li>
</ul>

g.css:
//common
@blue:blue;
@big:24px;
@middle:14px;
//module
.entry p,.list p{color:#000;font-size:12px;}
.list h2,.entry_title{margin:10px 20px;font-size:@big;color:@blue;}
.entry_counter{padding:0 10px;font-size:@middle;color:@blue;}
a{color:#000;}
[/html]

OAuth入门，让你的网站连接新浪微博

新浪微博拥有两种认证方式：OAuth、Http Basic Authoration。OAuth因为不需要将用户的密码存于第三方网站而更安全，被大多数web应用采用。不过Basic Authoration也因为更简单容易操作所以适合本地应用。

闲话少说，根据http://open.t.sina.com.cn/wiki/index.php/OAuth的描述。整个OAuth的流程如下：

要想让自己的网站与新浪微博链接起来，需要如下步骤（以土豆为例子）：

  1. 在登录页面放置“新浪微博按钮”，http://login.tudou.com/login.do?service=http://www.tudou.com/&noreg=ok；
  2. 在网站端提供跨域代理，获得包含oauth请求的token，如：http://login.tudou.com/xiaonei/sina/auth.action；
  3. 点击按钮，用iframe显示这个链接；
  4. 此链接会返回302响应，重定向到：http://api.t.sina.com.cn/oauth/authorize?oauth_token=33a3883d7c121c397b29755305f65a4a
  5. 这个链接会显示新浪的登录表单，用户登录后表示用户授权然后重定向到事先设置好的callback_url。如豆的链接是：http://login.tudou.com/xiaonei/sina/lo.action?oauth_token=475c264a22b3aae92b59046f1868c12f&oauth_verifier=173444；
  6. 这个链接会记录用户的登录状态，生成一个新的帐号，并且由js控制重新定向到土豆首页。土豆在这个页面的内容是：
    [js]
    <script>
    document.domain = ‘tudou.com’;
    (function() {
    if(window.syc) return;
    window.syc = true;
    parent.loginSina("http://login.tudou.com/login.do?act=newreg&user=_78484255&code=47g7j2mgafc71o0l4gj9&service=http://www.tudou.com&quot;);
    })();
    </script>
    [/js]

这样就完成了用新浪微博账户登录的流程。主要涉及到跨域的步骤是第一步和第六步。第一步使用服务器端做代理的模式避免了直接去跨域请求。第六步使用了iframe方式，通过设置domain实现了跨域。
