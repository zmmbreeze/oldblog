「感悟」拥抱HTML5

上个月参加了在上海参加了拥抱HTML5的会议，有很多不错的收获。这次会议一直全程做着笔记，结束之后一直想要写些东西总结一下这次会议的心得，但是无奈之前总是被各种各样的事情拖住，并且还一直疾病缠身。今天终于有空写些东西了。

  • HTML5的设计原理
  • HTML5现在的应用
  • HTML5的展望

HTML5的设计原理

整个会议中最有意义的一部分就是Jeremy Keith分享的HTML5设计原理，他是《Javascript DOM编程艺术》的作者。HTML5的标准制定者主要是原来两个阵营的人组成，这两个阵营是：XHTML2（W3C组织）和Web application 1.0（WHATWG，主要是由浏览器厂商组成）。如何让这两个思维和目标都不同的组织一起协同工作，产生一个新的标准这是一件困难的事情。这就需要有一个两方都同意的设计原则！

1. 避免不需要的复杂性

HTML5给人印象最深的就是它的doctype：

[html]<!DOCTYPE html>[/html]

非常的简洁！因为doctype的作用就是提供给HTML验证器使用的，大多数浏览器根本不在乎有没有这个doctype（IE需要有doctype才会使用标准模式绘制）。所以没有必要设置过份长的doctype，甚至连5都不需要。

与此相同的还有：

[html]
<meta charset="utf-8"/>

<link rel="stylesheet" href="file.css" />
<script src="ss.js"></script>

[/html]

这些都最大程度的让代码变得简洁，同时又支持所有现在的主流浏览器！

2. 支持已有的内容

HTML5的规范不像XHTML那样要求严格的闭合标签，用小写。实际上他支持现在的任何一种形式：

[html]<img src="foo" alt="bar" />
<p>Hello world</p>

<img src="foo" alt="bar">
<p>Hello world

<IMG SRC="foo" ALT="bar">
<P>Hello world</P>

<img src=foo alt=bar>
<p>Hello world</p>
[/html]

3. 解决真实的问题

HTML5的设计原则之一就是要设计出真正解决问题的标准，而不是标准制定者的自娱自乐。原来如下的代码在以前的标准中是不支持的：

[html]<a href="#asd">
<h2>XXXXXX</h2>
<p>PPPPPPPPPPPPPP</p>
</a>[/html]

现在支持了，因为所有的主流浏览器都支持，大家也都这么用，更重要的是它能解决问题，我们不用写两个a标签了！

4. 求真务实

HTML5给人印象最深的应该是这几个标签吧！

header\nav\section\article\aside\footer\details…
标准的制定者统计了现在网页上使用最频繁的ID，得到以上最常用的几个id，分别表示网页的不同部分。于是标准的设计者们，根据这些ID制定了上面几个新的标签。可见标准的制定不是拍脑袋想出来的，而是真正的从分析数据开始，以真实的数据作为依据来制定标准！

5. 优雅降级

使用了新功能，但是原来的浏览器不支持怎么办？这可能是前端甚至是所有的开发工程师都会遇到的难题：如何优雅的降级？HTML5标准在制定时也考虑到了这些问题，举个最明显的例子就是input表单。如下：

[html]
<input type="number" />
[/html]

如果旧浏览器的input不支持新的type类型number，那么它会自动退化成type=”text”，而不是直接忽略掉！

再举个例子，video标签：

[html]
<video>
<source src="movie.mp4">
<source src="movie.ogv">
<source src="movie.webm">
<object data="movie.swf">
<a href="movie.swf">XXXX</a>
</object>
</video>
[/html]

上面的代码做到了5层退化处理，首先是编码格式的三种判断，然后如果还不支持则使用flash播放！如果flash还不支持，则直接给出下载链接。

6. 最终用户优先

这可以说是所有设计原则中最重要的一部分，它将标准制定的考虑顺序分为：

a. 最终用户优先

b. 其次是开发人员（前端工程师）

c. 浏览器开发者

d. 标准制定者

这样可以避免HTML5重蹈XHTML2标准的覆辙，避免标准的制定变成闭门造车。也使得web页面对用户更加的友好，页面产生的错误将不会让最终用户看到！前端工程师作为次级优先的顺序，也可以对标准的制定起到很重要的作用。

也正式因为这个“最终用户优先”的设计原则，就可以看到HTML5的标准必然会付诸实际。因为浏览器的开发者所拥有的话语权高于标准的制定者。

HTML5现在的应用

这次的分享中，有两个session给我的印象比较深：一个是Web qq在HTML5方面的应用、另一个是优酷在iphone上的视频标签应用。这两个session是有个共同点：线上产品中应用HTML5！

打开Web qq的未来之门

Web qq一直是比较喜欢的产品，但是我一直没想到腾讯的工程师会在Web qq中应用这么多新技术：

1. canvas实现动态桌面，用了box2d 物理引擎
2. 支持ipad，使用了video标签
3. IE9新特性支持，新图标实现消息提醒，jump list，预览图的播放按钮
4. Notification实现
5. 位置服务
6. canvas实现时钟，但是作了svg和vml的跨浏览器实现

除了以上列举的技术之外，例如音乐盒子的实现用到了如下的技术：
1. HTML5 audio
2. CSS3 伪类选择器、RGBa色彩
3. 圆角、阴影、渐变
4. CSS33 动画

另外注意到一个细节，腾讯的工程师对于兼容性支持的顺序大都使用了如下的顺序：

html5 ==> flash实现 ==> 浏览器的非标准支持
个人认为这么做的好处如下：

1. 支持最新的API，让新浏览器用户得益！（同时可以保证开发人员与时共进）

2. 用flash实现是因为它能够最大限度的模拟，通常flash实现的会比浏览器的非标准方法要模拟的更好。另外flash的安装量的确很高。

3. 浏览器的非标准支持，可以让大部分低级无flash的浏览器也能体验到该特性

通过以上降级的方式基本就可以覆盖所有的用户。当然也会有浏览器非标准方法或flash无法模拟的情况，这就需要做好提示升级的准备。

HTML5在优酷中的应用

这是一个有很多干货的session。是优酷的前端工程师分享：优酷在支持ipad、iphone等设备时候的一些经验。

这其中也讲到了后端的一些分享，因为flash已经支持了H.264，所以目前后端存储的flash格式也已经变成了H.264的格式了。

目前优酷主要使用了H.264的video格式来支持iphone、ipad。但是他们还没有支持android平台的编码webm。

还谈到了ipad 1代中video标签的一些bug：
1. 自动播放（autoplay不能用）
2. 预加载（preload不能用）
3. 音量设置(不能用)

HTML5的展望

在谈到展望之前，先看一组数据：

53亿人拥有手机
2011年售出1.6亿
19%的智能手机
8亿人用智能手机上网，4年后会超越PC
其中26%的手机上网用户会玩游戏
2011年的移动设备出货量超越了桌面设备

可见，HTML5在移动平台上的普及速度会大大的超过PC平台。这意味着，我们可以在智能手机的前端开发中用到更多的HTML5技术。我们可以用HTML5来写手机web应用，也可以做基于phone gap的原生app，甚至是游戏的开发！总之移动平台会是HTML5的首个突破口。

在手机平台上用前端技术开发游戏可以说是很多人想做的事情！而且根据现在手机发展的速度，相信不久的将来会有更NB的手机出现，到时运行的速度不再是问题，再加上HTML5可以帮我们解决大部分的兼容性问题。web game可以说是非常有前途的发展方向。

当然在这之前还有几个困难要克服：

1. 声音（Audio支持很悲剧）

2. 网速（联通、移动要给力啊！）

3. 手机浏览器的兼容性（iphone怎么都去不掉浏览器的工具栏啊！）

另外对于手机游戏我得到的几个信息如下：

1. 目前canvas的性能还有待提高，但是iphone这样高端的机器已经很不错了

2. 目前来说CSS的方式实现游戏会比canvas效率要高

3. canvas相比CSS来说，兼容性更好。如果不是要求特别高帧数的游戏，推荐用canvas

4. 支持3D加速的浏览器，用3D API来做2D的事情会比原生的2D方法更快！（比较崎岖…）

 

 

当然还有很多有意思的分享，如“Sass & Compass”以及”web form 2.0”的介绍都很不错！

不过，还是就写这些了……
