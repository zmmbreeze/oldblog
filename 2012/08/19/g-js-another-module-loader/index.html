G.js: Another module loader

一晃就快两年了，两年前正式加入了果壳网做前端开发工作。那时的果壳还是一个纯html 1.0的页面，很少的ajax交互。后来交互慢慢变多，不过大多是为了后端性能问题（需要前端后加载，充分利用缓存等等原因）。中间有一段时间为了SEO优化ajax操作又变少了。不过最终推出的果壳问答、通知系统、可视化编辑器等功能或产品都是相对较重的。当时requirejs还刚刚兴起，看着十几k的大小和许多用不到的功能。我考虑了很久，为了减少下载文件大小同时保持现有代码的兼容性。于是开发了G.js来解决模块之间的依赖问题。之后果壳前端又有新人加入，于是我们又基于nodejs设计开发了配套的可视化构建工具guokr-build。

不得不承认G.js是一个重复的轮子，它削减了很多requirejs中的功能得到了1.7k（压缩后）大小的优势。这两年它的版本也从0.1变成了0.5，从很多其他优秀的框架中汲取了灵感。今年年初终于将它开源了。现在已经有了相当多优秀的开源模块加载器，如果你需要一个功能全面的模块加载器这里推荐seajs，1.2.0版只有11k大小相比requirejs瘦身了些且有很多不错的插件，开发也一直很活跃。

回头再来说下G.js，它也有自己优势。

小

压缩后只有1.7k，完全可以内嵌到html中减少请求数量。刚开始果壳没有使用任何CDN，所以对页面的js连接数量和大小都有限制。一直过着“巧妇难为无米之粥”的日子，故对大小特别敏感。

多模块

即是同一个文件中可以有多个模块。很多的模块加载器都是一个文件只能有一个模块，这算是当时想迁移到requirejs的最大阻碍了。旧代码中有很多多个模块放在一个文件中，一个人是无法完成这大量的修改的。于是实现了这个特性。

可视化工具

guokr-build是一个非常不错的构建工具，可以用于构建G.js的代码。理论上来说是任何代码，因为它是根据代码中书写的import原则来合并压缩代码的。

Javascript使用以下注释形式：

[js]//@import "a.js";
//@import "../b.js"; [/js]

CSS支持默认import方式：

[css]@import "a.css";
@import "../b.css";[/css]

劣势

说完优势该是劣势了：
  1. 不支持匿名模块，同时模块的命名规则较为复杂
  2. 需要手写@import规则才能使用构建工具，对于开发来说是个负担
  3. G.js实现的是类AMD的模块加载规范，但又不是完全支持

如何使用

这里只是举个例子，详细使用可以看README.md。

定义一个模块：

[js]
// file: lib/a.js
G.def(‘a’, function() {
return ‘a’; // G.req(‘a’) === ‘a’;
});
// file: lib/b.js
G.def(‘b’, { name: ‘b’ });
// file: lib/c.js
G.def(‘c’, [‘a’, ‘b’], function (a, b) {
return a + b.name; // G.req(‘c’) === ‘ab’;
});
[/js]

使用一个模块：

[js]
// file: demo.html
<script src="./G.js"></script>
<script>
G.req(‘c’, function (c) {
G.log(‘c’); // print ‘ab’ by console.log if exist
});
</script>
[/js]

文件目录:

./demo.html
./G.js
./lib/
     c.js
     b.js
     a.js

下载

G.js由于是专门针对guokr.com写的，所以使用了 G.req/G.def的api。如果你想用require/define的api那么可以下载修改版本。
