Javascript线程模型探究

Javascript是以实现浏览器上的异步操作而出名的一门浏览器端语言。例如如下的代码（基于jQuery）：

[js]
//do something 1
$.get("http://mzhoublog.appspot.com",function(data){
//do something after get the web
console.log(data);
},"text");
//do something 2
[/js]

以上代码执行顺序是：

[js]
do something 1
do something 2
console.log(data);
[/js]

这体现了javascript的异步特性，但是究竟javascript支不支持异步特性。先看如下代码：

[js]setTimeout(function(){
console.log("setTimeout;");
},0);
while(true){}
console.log("while over");
[/js]

最后什么都没有打印！进入死循环后打印不出while over这很正常，但是为什么打印不出setTimeout呢？首先看setTimeout函数的作用：用于在n毫秒之后执行指定的函数。那么0毫秒就是立即执行，为什么没有打印呢？先看如下一张图（浏览器的线程模型）：

很明显从图中来看，javascript是纯粹的单线程模型。任何时刻每个页面都只有一个线程被用于执行javascript。这就能解释为什么setTimeout没有被打印的原因了！

因为setTimeout函数所做的是将传入的函数和时间发送给浏览器的定时器，当定时器发现时间到是便触发，将函数置入到浏览器javascript执行线程的最末端。因为是单线程，前面进入了死循环，函数自然执行不了！

ajax的异步获取是怎么实现的呢？

和setTimeout函数类似。当执行异步获取时，浏览器会新建一个线程执行异步获取。如果设置了回调函数则会在获取之后，将函数置入javascript执行线程的最末端。

这样差不多理解了浏览器的线程模型！不过还有一个浏览器GUI渲染线程，这个Javascript执行线程有什么关系呢？

[js]&lt;!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd"&gt;
&lt;html lang="en"&gt;
&lt;head&gt;
&lt;meta http-equiv="Content-Type" content="text/html;charset=UTF-8"&gt;
&lt;title&gt;&lt;/title&gt;
&lt;script type="text/javascript"&gt;
function run() {
while(true) {console.log("ss")}
}
&lt;/script&gt;
&lt;/head&gt;
&lt;body&gt;
&lt;input type="text"&gt;ss&lt;/input&gt;
&lt;input type="button" value="start" onclick="run();"/&gt;
&lt;/body&gt;
&lt;/html&gt;[/js]

将上面代码保存运行后，点击start按钮，浏览器就会进入假死状态！因为run函数中有死循环，所以js一直在执行！而javascirpt执行线程与gui渲染线程是互斥的！因为它们都需要访问dom树！如果在js修改dom树的同时gui在渲染会导致界面产生不一致！

那么如何避免浏览器的假死呢？

对长时间操作进行分时执行的操作，例如如下代码：

[js]for(var i=0; i &lt; items.length; i++){
process(items[i]);
}[/js]

假设以上代码多次process的执行导致需要很长的执行时间，这很容易导致浏览器的假死。这种情况经常发生在有很多表格需要进行异步处理的界面中。

如何进行分时处理呢？

[js]function chunk(array, process, context){
var items = array.concat(); //clone the array
setTimeout(function(){
var item = items.shift();
process.call(context, item);

    if (items.length &amp;gt; 0){
        setTimeout(arguments.callee, 100);
    }
}, 100);

}[/js]

以上代码都取自于牛人Nicholas C. Zakas的blog文章：Speed up your JavaScript, Part 1，他又在他后来的文章Timed array processing in JavaScript中做了更深入的阐述。并提出了如下的改良方案：

[js]//Copyright 2009 Nicholas C. Zakas. All rights reserved.
//MIT Licensed
function timedChunk(items, process, context, callback){
var todo = items.concat(); //create a clone of the original

setTimeout(function(){
    var start = +new Date();

    do {
         process.call(context, todo.shift());
    } while (todo.length &amp;gt; 0 &amp;amp;&amp;amp; (+new Date() - start &amp;lt; 50));

    if (todo.length &amp;gt; 0){
        setTimeout(arguments.callee, 25);
    } else {
        callback(items);
    }
}, 25);

}[/js]

国内有人总结了文章的几个要点如下：

setTimeout 延时 25ms, 是因为浏览器的时间分辨率问题。25ms 可以保证主流浏览器都顺畅（有喘息的机会去更新 UI）。

顺便八卦：Safari 4 和 Chrome 2 真不是吹的，速度嗖嗖的。相比，Firefox 和 IE 慢得够可以。

最后，很钦佩 Nicholas C. Zakas. 短短的一段代码，蕴含了博学的可用性知识 + 专业的 JS 技能 + 细心的疯狂测试。如此牛人，实在是学习的绝佳榜样。
